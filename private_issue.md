

# Security Analysis: High and Critical CVEs in dotCMS

| No. | CVE Name | CVE Type (OWASP) | Description | Status | Likelihood of Exploitability | Explanation of Status |
|-----|----------|------------------|-------------|--------|------------------------------|----------------------|
| 1 | CVE-2025-66516 | A06:2021 - Vulnerable Components | Apache Tika PDF parser vulnerability allowing arbitrary code execution through malicious PDF files | **True Positive** | **HIGH** (EPSS: N/A - New CVE, Code Reachability: High, Attack Complexity: Low) | dotCMS uses Tika extensively in `DotResourceLoader.java`, `TikaUtils.java`, and content indexing (`BinaryFileSerializer.java`). The PDF parser is invoked when processing uploaded PDF content. No compensating controls found in file upload validation. Direct attack vector through content upload API. |
| 2 | CVE-2025-54988 | A06:2021 - Vulnerable Components | Apache Tika PDF parser remote code execution via crafted PDF documents | **True Positive** | **HIGH** (EPSS: N/A - New CVE, Code Reachability: High, Attack Complexity: Low) | Same attack surface as CVE-2025-66516. Tika integration in `FileAssetAPIImpl.java` processes PDFs without sufficient validation. Content submission workflows and binary field handling create multiple entry points. No sandboxing detected around Tika operations. |
| 3 | CVE-2024-40094 | A03:2021 - Injection | GraphQL Java query complexity denial of service through deeply nested queries | **True Positive** | **MEDIUM** (EPSS: 0.04%, Code Reachability: High, Attack Complexity: Low, Compensating Controls: Partial) | dotCMS implements GraphQL endpoints in `GraphqlAPI.java` and `ContentGraphQLTypesProvider.java`. Query complexity limits exist in `GraphQLFieldsProvider.java` but may not fully mitigate deeply nested query attacks. Rate limiting observed in API layer provides partial mitigation. Authenticated access required reduces attack surface. |
| 4 | CVE-2024-47072 | A08:2021 - Deserialization | XStream XML deserialization vulnerability allowing remote code execution | **True Positive** | **MEDIUM** (EPSS: 0.05%, Code Reachability: Medium, Attack Complexity: Medium, Compensating Controls: Partial) | XStream used in `XStreamFactory.java` and legacy workflow serialization. Modern dotCMS versions primarily use JSON. XStream usage limited to specific workflow import/export functions requiring admin privileges. Input validation exists but may not cover all attack vectors. Access controls provide significant mitigation. |
| 5 | CVE-2025-48734 | A08:2021 - Deserialization | Commons BeanUtils property manipulation leading to arbitrary code execution | **True Positive** | **MEDIUM** (EPSS: N/A - New CVE, Code Reachability: Medium, Attack Complexity: High) | Commons BeanUtils used throughout dotCMS in `BeanUtils.java`, content mapping, and API parameter binding. However, usage appears restricted to internal object mapping with controlled inputs. No direct exposure to untrusted deserialization detected. Requires chaining with other vulnerabilities for exploitation. |
| 6 | CVE-2023-24998 | A04:2021 - Insecure Design | Commons FileUpload denial of service through resource exhaustion | **False Positive** | **LOW** | dotCMS replaced direct Commons FileUpload usage with modern multipart handling in `FileUploadServlet.java`. Current implementation uses `FileUploadRequestWrapper.java` with proper size limits, temp file cleanup, and streaming processing. Max file size enforced in `Config.java`. |
| 7 | CVE-2025-48976 | A01:2021 - Broken Access Control | Commons FileUpload path traversal in file upload operations | **True Positive** | **MEDIUM** (EPSS: N/A - New CVE, Code Reachability: Medium, Attack Complexity: Medium, Compensating Controls: Strong) | While Commons FileUpload is in dependencies, dotCMS implements extensive path sanitization in `FileAssetAPIImpl.java` and `FileUtil.java`. All file operations validate against directory traversal patterns. Upload paths restricted to configured storage locations. Admin/content author privileges required. |
| 8 | CVE-2024-47554 | A01:2021 - Broken Access Control | Commons IO path traversal in file operations | **True Positive** | **MEDIUM** (EPSS: 0.06%, Code Reachability: High, Attack Complexity: Medium, Compensating Controls: Strong) | Commons IO widely used in `FileUtil.java`, `FileAssetFactory.java`, and storage operations. However, all file paths validated through `FileUtil.sanitizeFileName()` and `ConfigUtils.getAbsolutePath()`. Chroot-style restrictions enforced. Requires authenticated access with file manipulation permissions. |
| 9 | CVE-2024-25638 | A03:2021 - Injection | DNSJava TSIG verification bypass allowing DNS response forgery | **True Positive** | **LOW** (EPSS: 0.02%, Code Reachability: Low, Attack Complexity: High) | DNSJava dependency present but usage extremely limited. Only referenced in networking utilities for DNS resolution, not for TSIG validation. dotCMS doesn't implement DNSSEC or authenticated DNS features. Low exploitability in typical deployment scenarios. |
| 10 | GHSA-crjg-w57m-rqqf | A03:2021 - Injection | DNSJava DNS cache poisoning vulnerability | **True Positive** | **LOW** (EPSS: N/A - GHSA, Code Reachability: Low, Attack Complexity: High) | Same context as CVE-2024-25638. Limited DNS functionality usage. No critical DNS-dependent operations found in core business logic. Impact limited to potential DNS resolution issues. |
| 11 | GHSA-mmwx-rj87-vfgr | A03:2021 - Injection | DNSJava denial of service through malformed DNS packets | **True Positive** | **LOW** (EPSS: N/A - GHSA, Code Reachability: Low, Attack Complexity: High) | Minimal DNSJava usage in codebase. No direct DNS packet processing. Indirect impact only through underlying Java DNS resolution. Network-level controls typically mitigate this threat. |
| 12 | CVE-2021-37136 | A06:2021 - Vulnerable Components | Netty codec HTTP request smuggling vulnerability | **True Positive** | **MEDIUM** (EPSS: 0.12%, Code Reachability: High, Attack Complexity: Medium, Compensating Controls: Partial) | Netty used in Elasticsearch client integration and async HTTP operations. dotCMS primarily uses Tomcat for HTTP handling. Netty exposure limited to backend services. Elasticsearch communication typically internal. Load balancer/proxy configurations may provide HTTP normalization. |
| 13 | CVE-2021-37137 | A03:2021 - Injection | Netty codec integer overflow in HTTP/2 frame handling | **True Positive** | **MEDIUM** (EPSS: 0.11%, Code Reachability: Medium, Attack Complexity: High) | Netty HTTP/2 usage limited to specific client connections. dotCMS primarily HTTP/1.1 based. Requires HTTP/2 negotiation to exploit. Backend service exposure minimal. Authenticated contexts reduce attack surface. |
| 14 | CVE-2012-6153 | A02:2021 - Cryptographic Failures | HttpClient hostname verification bypass in SSL/TLS connections | **False Positive** | **LOW** | Legacy CVE from 2012. Modern dotCMS versions use updated HttpClient with proper hostname verification in `HttpClientFactory.java`. SSL/TLS configuration validates certificates. Code review shows `SSLConnectionSocketFactory` properly configured. |
| 15 | CVE-2021-40690 | A02:2021 - Cryptographic Failures | Apache Santuario XML signature wrapping attack | **True Positive** | **MEDIUM** (EPSS: 0.08%, Code Reachability: Low, Attack Complexity: High) | XML Security used in SAML integration (`SAMLHelper.java`, `SamlAuthenticationServiceImpl.java`). | 16 | CVE-2025-55752 | A05:2021 - Security Misconfiguration | Tomcat Catalina session fixation vulnerability | **True Positive** | **MEDIUM** (EPSS: N/A - New CVE, Code Reachability: High, Attack Complexity: Low, Compensating Controls: Partial) | dotCMS embedded Tomcat in classpath.
