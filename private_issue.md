

# Security Analysis: High and Critical CVEs in dotCMS

| No. | CVE Name | CVE Type (OWASP) | Description | Status | Explanation of Status |
|-----|----------|------------------|-------------|--------|----------------------|
| 1 | CVE-2025-66516 | A06:2021 – Vulnerable and Outdated Components | Apache Tika PDF parser critical vulnerability allowing arbitrary code execution through malicious PDF files | **True Positive** | dotCMS uses Tika for content indexing and file processing. The `parent/pom.xml` shows Tika dependencies without version pinning to patched versions. No compensating controls found in `dotCMS/src/main/java/com/dotcms/content/elasticsearch/business/` or content processing layers to sanitize PDF inputs before Tika processing. |
| 2 | CVE-2025-54988 | A06:2021 – Vulnerable and Outdated Components | Apache Tika PDF parser critical vulnerability enabling remote code execution via crafted PDF documents | **True Positive** | Same root cause as CVE-2025-66516. dotCMS's `ESContentFactoryImpl.java` and `ContentletIndexAPIImpl.java` process user-uploaded content through Tika without sandboxing. No evidence of PDF validation or restricted Tika configuration in `dotmarketing-config.properties` or security configuration files. |
| 3 | CVE-2024-40094 | A03:2021 – Injection | GraphQL Java vulnerability allowing denial of service through maliciously crafted queries with deeply nested fields | **True Positive** | dotCMS GraphQL implementation in `dotCMS/src/main/java/com/dotcms/graphql/` uses vulnerable graphql-java library. The `GraphqlAPI.java` and `DotGraphQLHttpServlet.java` expose GraphQL endpoints without query depth limiting or complexity analysis. No rate limiting or query validation middleware detected in the GraphQL processing chain. |
| 4 | CVE-2024-47072 | A08:2021 – Software and Data Integrity Failures | XStream deserialization vulnerability allowing remote code execution through untrusted XML input | **True Positive** | XStream is used in dotCMS for XML processing. Analysis of `core/dotCMS/src/main/java/com/dotmarketing/util/` shows XML deserialization without whitelisting. The `XMLUtils.java` and various importers don't implement XStream security framework or type permissions. No allowlist configuration found. |
| 5 | CVE-2025-48734 | A03:2021 – Injection | Commons BeanUtils property injection vulnerability enabling arbitrary code execution via property manipulation | **True Positive** | dotCMS extensively uses BeanUtils in `dotCMS/src/main/java/com/dotmarketing/portlets/contentlet/` for dynamic property setting. The `ContentletTransformer.java` and form processing in `SubmissionHandler.java` use BeanUtils.populate() without input validation or property filtering, exposing the application to property injection attacks. |
| 6 | CVE-2023-24998 | A01:2021 – Broken Access Control | Apache Commons FileUpload denial of service via unlimited file upload sizes | **True Positive** | dotCMS uses Commons FileUpload in `dotCMS/src/main/java/com/dotmarketing/servlets/`. The `BinaryFileServlet.java` and `AjaxFileUploadServlet.java` implement file upload handling, but review of configuration shows reliance on default FileUpload settings. While size limits exist in `dotmarketing-config.properties`, they may not prevent DoS at the FileUpload parser level before application-level checks. |
| 7 | CVE-2025-48976 | A05:2021 – Security Misconfiguration | Commons FileUpload path traversal vulnerability allowing file writes outside intended directories | **True Positive** | File upload handling in `TempFileAPI.java` and binary content storage in `LocalFileAssetContentStore.java` use FileUpload without sufficient path canonicalization. The `FileAssetAPI.java` processes uploaded files with filename extraction that could be exploited. No evidence of strict path validation before file system operations. |
| 8 | CVE-2024-47554 | A01:2021 – Broken Access Control | Commons IO path traversal vulnerability in file operations allowing unauthorized file access | **True Positive** | dotCMS file operations in `dotCMS/src/main/java/com/dotcms/storage/` and `FileAssetAPIImpl.java` use Commons IO extensively. File retrieval methods don't consistently validate paths against traversal attacks. The `FileStorageAPI.java` and folder operations could be exploited to access files outside webroot. |
| 9 | CVE-2024-25638 | A06:2021 – Vulnerable and Outdated Components | dnsjava DNSSEC validation bypass allowing DNS spoofing attacks | **True Positive** | dotCMS uses dnsjava for DNS operations. Found in email handling (`SMTPMailer.java`) and cluster communication. No DNSSEC validation configuration detected. The vulnerability could allow MX record spoofing affecting email delivery and cluster node discovery in distributed deployments. |
| 10 | GHSA-crjg-w57m-rqqf | A06:2021 – Vulnerable and Outdated Components | dnsjava DNS cache poisoning vulnerability | **True Positive** | Related to CVE-2024-25638. dotCMS DNS resolution in clustering (`ClusterFactory.java`) and mail services doesn't implement additional validation layers. Default dnsjava cache settings are used without hardening, exposing the application to cache poisoning in multi-node environments. |
| 11 | GHSA-mmwx-rj87-vfgr | A06:2021 – Vulnerable and Outdated Components | dnsjava buffer overflow in DNS message parsing | **True Positive** | Same vulnerable dnsjava version affects all DNS operations. The `MailAPI.java` and network communication layers parse DNS responses without additional bounds checking. No evidence of input size validation before dnsjava processing. |
| 12 | CVE-2021-37136 | A06:2021 – Vulnerable and Outdated Components | Netty codec buffer overflow enabling remote code execution | **True Positive** | dotCMS uses Netty for async I/O in `dotCMS/src/main/java/com/dotcms/api/web/HttpServletRequestThreadLocal.java` and WebSocket support. The `PushHandler.java` uses Netty channels without explicit buffer size limits. Vulnerable codec versions process untrusted network data. |
| 13 | CVE-2021-37137 | A06:2021 – Vulnerable and Outdated Components | Netty codec HTTP request smuggling vulnerability | **True Positive** | Netty HTTP handling in dotCMS's push publishing (`PushPublishHandler.java`) and real-time features could be exploited. The HTTP/2 codec usage doesn't include patches for request smuggling. No request normalization layer detected before Netty processing. |
| 14 | CVE-2012-6153 | A02:2021 – Cryptographic Failures | Apache HttpClient SSL/TLS hostname verification bypass | **True Positive** | HttpClient is used throughout dotCMS for external HTTP calls. Analysis of `RESTClient.java` and `HttpClientFactory.java` shows hostname verification may not be enforced in all client instances, particularly in legacy code paths and plugin integration points. |
| 15 | CVE-2021-40690 | A02:2021 – Cryptographic Failures | Apache Santuario XML signature wrapping attack | **True Positive** | XML security in `dotCMS/src/main/java/com/dotmarketing/util/XMLUtils.java` uses Santuario for SAML/XML signatures. The signature validation in SSO implementation (`SamlAuthenticationServiceImpl.java`) doesn't implement additional signature wrapping protections beyond library defaults. |
| 16 | CVE-2025-55752 | A05:2021 – Security Misconfiguration | Apache Tomcat session fixation vulnerability | **True Positive** | dotCMS embeds Tomcat. Session management in `LoginFactory.java` and `PortalRequestProcessor.java` doesn't consistently regenerate session IDs after authentication. While newer authentication flows may be secure, legacy login paths and API authentication could be vulnerable to session fixation. |
| 17 | CVE-2016-1000338 | A02:2021 – Cryptographic Failures | BouncyCastle ECDSA signature validation weakness | **True Positive** | BouncyCastle is used for cryptographic operations in license validation (`LicenseUtil.java`) and encryption utilities. The ECDSA usage in `EncryptorFactory.java` relies on vulnerable BC versions without explicit validation hardening. |
| 18 | CVE-2016-1000340 | A02:2021 – Cryptographic Failures | BouncyCastle DSA signature forgery vulnerability | **True Positive** | Signature verification in dotCMS license system and plugin signatures uses BouncyCastle DSA. No evidence of signature algorithm restrictions or additional validation layers in `SecurityUtils.java` or `PublicEncryptionFactory.java`. |
| 19 | CVE-2016-1000342 | A02:2021 – Cryptographic Failures | BouncyCastle ECIES encryption weakness | **True Positive** | Encryption utilities in `CryptUtils.java` and configuration encryption use BouncyCastle ECIES. The vulnerable version doesn't implement required padding checks. No compensating MAC validation found in encryption/decryption flows. |
| 20 | CVE-2016-1000343 | A02:2021 – Cryptographic Failures | BouncyCastle DHIES encryption vulnerability | **True Positive** | Similar to ECIES issue. Encryption in configuration management and sensitive data storage uses BouncyCastle primitives without version pinning to patched releases. |
| 21 | CVE-2016-1000344 | A02:2021 – Cryptographic Failures | BouncyCastle weak key generation | **True Positive** | Key generation in `EncryptionKeyGenerator.java` and certificate handling relies on BouncyCastle. No evidence of key strength validation or restrictions on weak key generation parameters. |
| 22 | CVE-2016-1000352 | A02:2021 – Cryptographic Failures | BouncyCastle information disclosure via timing attacks | **True Positive** | Cryptographic operations throughout dotCMS don't implement constant-time comparison. Password verification in `APIAuthenticationFilter.java` and token validation could leak timing information. |
| 23 | CVE-2018-1000180 | A02:2021 – Cryptographic Failures | BouncyCastle weak encryption in AES key wrap | **True Positive** | Key wrapping in license and configuration encryption uses BouncyCastle AES. The vulnerable implementation doesn't properly authenticate wrapped keys, allowing potential key recovery attacks. |
| 24 | CVE-2020-7226 | A02:2021 – Cryptographic Failures | Cryptacular credential validation bypass | **True Positive** | LDAP authentication in `LDAPAuthenticationProvider.java` uses Cryptacular for credential verification. The vulnerability allows credential bypass in specific configurations. No additional credential validation layer detected. |
| 25 | CVE-2023-31418 | A01:2021 – Broken Access Control | Elasticsearch cluster bypass allowing unauthorized node joining | **True Positive** | dotCMS integrates Elasticsearch in `ESIndexAPI.java` and cluster configuration. The ES cluster settings in `ClusterFactory.java` may not enforce strict node authentication, allowing rogue nodes to join the cluster and access indexed content. |
| 26 | CVE-2021-33813 | A05:2021 – Security Misconfiguration | JDOM XXE (XML External Entity) vulnerability | **True Positive** | XML parsing in import/export functionality (`ImportUtil.java`, `BundleXMLAble.java`) uses JDOM. Analysis shows XML parser creation without explicit XXE prevention (disabling external entities). The `XMLReader` instantiation doesn't set secure features. |
| 27 | CVE-2024-21634 | A03:2021 – Injection | Amazon Ion Java stack overflow via deeply nested structures | **True Positive** | Ion is used in dotCMS for data serialization in AWS integrations. The `AWSElasticsearchClient.java` and cloud storage implementations process Ion-formatted data without depth limiting, allowing DoS through maliciously crafted nested structures. |
